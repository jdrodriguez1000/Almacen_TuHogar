## 1. Resumen Ejecutivo

La Etapa 1.1 establece los cimientos de gobernanza, estructura y repositorio del proyecto **Dashboard de Inteligencia de Negocio para Almacén MultiTodo**. No se construye código de producto en esta etapa; se construye el entorno controlado en el que todo el desarrollo posterior ocurrirá de forma ordenada, trazable y sin ambigüedad.

**QUÉ se construye:** El conjunto de documentos de gobernanza, estructura de carpetas, repositorio Git inicializado y el estado de handoff que permitirá a cualquier agente o desarrollador retomar el proyecto sin pérdida de contexto.

**PARA QUIÉN:** Principalmente para el equipo técnico de Triple S y los agentes de IA que participan en el desarrollo. Secundariamente, sirve como evidencia de profesionalismo y orden para el cliente Almacén MultiTodo.

**POR QUÉ:** Sin gobernanza, cada decisión técnica se toma ad hoc, generando inconsistencias. Sin estructura de repositorio, los artefactos se pierden o colisionan. Sin handoff documentado, el contexto del proyecto muere con cada sesión. Esta etapa elimina esos riesgos desde el día cero.

**Problema que resuelve:** La ausencia de un marco de trabajo formal que garantice que el equipo, los agentes y el cliente operen bajo las mismas reglas, convenciones y entendimientos a lo largo de un proyecto de 3 meses.

---

## 2. Objetivos de Negocio

- `[OBJ-01]` Establecer un entorno de repositorio Git estructurado que soporte el ciclo de vida completo del proyecto (pipeline Python, dashboard Next.js, documentación SDD) sin conflictos de estructura.
- `[OBJ-02]` Definir y comunicar las leyes de comportamiento para todos los agentes de IA que interactúan con el repositorio, garantizando autonomía controlada y trazabilidad.
- `[OBJ-03]` Crear el estado de handoff inicial que permita retomar el proyecto en cualquier sesión futura sin pérdida de contexto ni necesidad de re-exploración.
- `[OBJ-04]` Asegurar que el scope del proyecto esté capturado, aprobado y vinculado a la estructura documental antes de que comience cualquier trabajo técnico.

---

## 3. Alcance

### 3.1 En Alcance

- `[REQ-01]` Crear y configurar el repositorio Git con rama `main` y estrategia de ramas `feat/*` documentada.
- `[REQ-02]` Crear `CLAUDE.md` con reglas de comportamiento, límites de autonomía, estándares de código, convenciones de idioma y protocolo de inicio/cierre de sesión.
- `[REQ-03]` Crear `PROJECT_scope.md` con alcance, objetivos, entregables, usuarios, criterios de éxito, restricciones y supuestos del proyecto completo.
- `[REQ-04]` Crear `PROJECT_handoff.md` con el estado macro inicial, etapa activa y próxima acción concreta.
- `[REQ-05]` Crear la estructura de carpetas del repositorio según la convención definida en `CLAUDE.md` (`pipeline/`, `web/`, `docs/` con todas sus subcarpetas).
- `[REQ-06]` Crear `docs/lessons/lessons-learned.md` con la sección inicial de la Etapa 1.1.
- `[REQ-07]` Crear `.claude/skill-router.md` con el mapa de agentes y skills disponibles para el proyecto.
- `[REQ-08]` Crear `.gitignore` con exclusiones para `.venv`, variables de entorno, artefactos DVC, y archivos de sistema.
- `[REQ-09]` Crear el commit inicial en `main` con todos los artefactos de gobernanza bajo el formato de commit definido.

### 3.2 Fuera de Alcance

- Configuración de Supabase, conexiones a base de datos o esquemas SQL (Etapa 1.2).
- Definición formal del Data Contract (Etapa 1.3).
- Cualquier línea de código de pipeline Python o componente Next.js.
- Configuración de CI/CD (GitHub Actions): se establece en etapas posteriores cuando haya código que testear.
- Instalación de dependencias (`requirements.txt`, `package.json`): se definen cuando los módulos sean implementados.
- Configuración de DVC: se activa cuando existan datasets que versionar.

## 7. Riesgos y Supuestos

| ID | Descripción | Probabilidad | Mitigación |
|---|---|---|---|
| `[RSK-01]` | `PROJECT_scope.md` incompleto o con ambigüedades que bloquen etapas posteriores | Media | Revisión explícita del scope con el cliente antes de marcar `Estado: Aprobado`; usar la convención de conflictos de CLAUDE.md si surge contradicción posterior |
| `[RSK-02]` | `CLAUDE.md` con reglas insuficientes que lleven a comportamiento inconsistente de agentes en etapas futuras | Baja | Revisión post-etapa 1.1 durante lessons-learned; actualización vía Control de Cambios si se detecta gap |
| `[RSK-03]` | Estructura de carpetas que no anticipe necesidades de etapas 3.x o 4.x | Baja | Estructura diseñada con base en el scope completo del proyecto; cambios vía CC si surge necesidad no anticipada |
| `[RSK-04]` | Commit inicial con artefactos incompletos que generen estado inconsistente en `main` | Baja | Checklist de cierre verificado antes del commit; auditoría con `/stage-audit f01_01` antes de cerrar |
