## 1. Resumen Ejecutivo

La Etapa 1.2 verifica que la infraestructura de base de datos en Supabase está correctamente configurada y lista para soportar el pipeline de datos del proyecto. No se construye lógica de negocio en esta etapa; se confirma que los cimientos de datos existen, son accesibles y tienen la forma esperada.

**QUÉ se construye:** Un conjunto de scripts de verificación y un documento de esquema (`docs/database/schema.sql`) que certifica que las tablas del cliente (`usr_*`) y las tablas de Triple S (`tss_*`) existen en Supabase con la estructura correcta, que los permisos RLS están activos, que los índices necesarios están presentes, y que el pipeline Python puede conectarse y operar contra la base de datos real.

**PARA QUIÉN:** Para el equipo técnico de Triple S. Los resultados de esta etapa son el prerrequisito directo para que la Etapa 1.3 (Data Contract) y las etapas de ingeniería de datos (Fase 3) operen sobre una base verificada.

**POR QUÉ:** Si las tablas no existen, tienen columnas incorrectas, o el pipeline no puede conectarse con las credenciales correctas, toda la cadena downstream falla silenciosamente o con errores difíciles de diagnosticar. Esta etapa elimina ese riesgo antes de que comience cualquier desarrollo de lógica de negocio.

**Problema que resuelve:** La incertidumbre sobre si la infraestructura de Supabase está lista para recibir el trabajo de las etapas 1.3 a 3.6. Sin verificación formal, los gaps de infraestructura se descubren tardíamente, cuando corregirlos tiene mayor costo.

---

## 2. Objetivos de Negocio

- `[OBJ-01]` Confirmar que las cuatro tablas del cliente (`usr_ventas`, `usr_inventario`, `usr_productos`, `usr_sedes`) existen en Supabase con la estructura de columnas, tipos de dato y constraints que el pipeline necesita para operar.
- `[OBJ-02]` Verificar que las tablas de infraestructura de Triple S (`tss_*`) — Bronze, Silver, Gold, logs, cuarentena — existen con el esquema correcto o crear las que falten mediante DDL documentado.
- `[OBJ-03]` Confirmar que el pipeline Python puede conectarse a Supabase usando las credenciales de entorno (`.env`) y ejecutar operaciones de lectura sobre `usr_*` y lectura/escritura sobre `tss_*`.
- `[OBJ-04]` Documentar el estado real del esquema en `docs/database/schema.sql` como fuente de verdad sincronizada con Supabase, que todas las etapas posteriores leerán antes de implementar.
- `[OBJ-05]` Validar que los permisos RLS (Row Level Security) y los índices de performance están activos en las tablas críticas para el ciclo diario del pipeline.

## 3. Alcance

### 3.1 En Alcance

- `[REQ-01]` Verificar existencia y estructura de `usr_ventas`: columnas, tipos, constraints y claves foráneas esperadas.
- `[REQ-02]` Verificar existencia y estructura de `usr_inventario`: columnas, tipos, constraints y claves foráneas esperadas.
- `[REQ-03]` Verificar existencia y estructura de `usr_productos`: columnas, tipos, constraints (especialmente el catálogo de ~100 SKUs).
- `[REQ-04]` Verificar existencia y estructura de `usr_sedes`: columnas, tipos, constraints (especialmente las 7 sedes activas).
- `[REQ-05]` Crear o verificar tablas Bronze: `tss_bronze_ventas`, `tss_bronze_inventario`.
- `[REQ-06]` Crear o verificar tablas Silver: `tss_silver_ventas`, `tss_silver_inventario`.
- `[REQ-07]` Crear o verificar tablas Gold: `tss_gold_daily_sales`, `tss_gold_abc_ranking`, `tss_gold_alerts`.
- `[REQ-08]` Crear o verificar tablas de soporte: `tss_pipeline_log`, `tss_cuarentena_ventas`, `tss_cuarentena_inventario`.
- `[REQ-09]` Verificar que RLS está habilitado en las tablas `tss_*` y que el `service_role` de Triple S tiene acceso completo (lectura/escritura).
- `[REQ-10]` Verificar que las tablas `usr_*` tienen RLS configurado para permitir solo lectura al `service_role` de Triple S.
- `[REQ-11]` Verificar o crear índices de performance en columnas de alta frecuencia de consulta: `fecha_hora` en `usr_ventas`, `sku` en `usr_ventas` e `usr_inventario`, `id_sede` en `usr_ventas` e `usr_inventario`.
- `[REQ-12]` Escribir script de conectividad Python (`pipeline/tests/test_infra_connectivity.py`) que verifica la conexión a Supabase usando `SUPABASE_URL` y `SUPABASE_SERVICE_KEY` del `.env` y ejecuta una consulta `SELECT 1` contra cada tabla.
- `[REQ-13]` Documentar el esquema completo verificado en `docs/database/schema.sql` como DDL reproducible (CREATE TABLE IF NOT EXISTS) sincronizado con el estado real de Supabase.

### 3.2 Fuera de Alcance

- Definición del Data Contract formal (Etapa 1.3): esta etapa verifica estructura, no reglas de negocio sobre los datos.
- Poblado de datos de prueba en ninguna tabla — solo se verifica estructura y conectividad.
- Implementación de validaciones Pandera (Etapa 3.1).
- Configuración de triggers de base de datos para lógica de negocio — solo se verifican los existentes, si los hay.
- Configuración de Supabase Auth para el dashboard web (Fase 3.6).
- Cualquier lógica de pipeline ETL (Fases 3.1–3.3).
- Ajustes o cambios a las tablas `usr_*` del cliente — son de solo lectura para Triple S.

## 7. Riesgos y Supuestos

| ID | Descripción | Probabilidad | Mitigación |
|---|---|---|---|
| `[RSK-01]` | Las credenciales de Supabase (`SUPABASE_URL`, `SUPABASE_SERVICE_KEY`) no están disponibles al iniciar la etapa, bloqueando toda verificación de conectividad | Alta | Identificar este prerequisito antes de iniciar la etapa; si no están disponibles, documentar el bloqueo en `PROJECT_handoff.md` y escalar al responsable del proyecto. Las tareas de verificación de esquema dependen de este acceso. |
| `[RSK-02]` | Las tablas `usr_*` del cliente no existen o tienen nombres diferentes a los asumidos (`usr_ventas`, `usr_inventario`, `usr_productos`, `usr_sedes`) | Media | Verificar con el cliente los nombres exactos de sus tablas antes de ejecutar los scripts. Si difieren, actualizar la nomenclatura en todos los documentos SDD mediante CC antes de continuar. |
| `[RSK-03]` | Las tablas `usr_*` existen pero con columnas faltantes o tipos de dato incompatibles (ej. fechas como TEXT en lugar de TIMESTAMPTZ) | Media | Documentar las discrepancias en `schema.sql` con comentarios `-- DISCREPANCIA`. Abrir CC para definir plan de corrección con el cliente antes de avanzar a Etapa 1.3. |
| `[RSK-04]` | RLS no está configurado en las tablas `usr_*`, exponiendo datos del cliente | Media | Si se detecta RLS deshabilitado en tablas del cliente, reportar inmediatamente al cliente como riesgo de seguridad. No modificar las tablas `usr_*` — solo documentar y notificar. |
| `[RSK-05]` | El `service_role` de Triple S tiene permisos insuficientes sobre las tablas `tss_*` o no tiene acceso de lectura a `usr_*` | Media | Verificar permisos al inicio de la etapa con una consulta simple. Si hay restricciones, coordinar con el administrador del proyecto Supabase para ajustar policies antes de continuar. |
| `[RSK-06]` | `docs/database/schema.sql` queda desincronizado de Supabase si el cliente modifica sus tablas `usr_*` sin notificar | Baja | Establecer comunicación formal con el cliente: cualquier cambio a `usr_*` debe notificarse a Triple S. El protocolo de control de cambios (CC) se activa si hay modificaciones post-cierre de la etapa. |
